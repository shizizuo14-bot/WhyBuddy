# 架构闭环评审 · 接缝问题与优化建议

> 审查对象：本目录下 6 张完整架构图（RBAC / 工作流 / 数据中台 / 页面设计器 / 应用中心 / AIGC 中台）。
> 审查视角：把 6 张图当成「同一个平台的 6 个视图」（共用 RBAC、共用数据模型、共用应用中心）。
> 结论：**每个系统"自己内部"的闭环都成立、分层干净；真正的问题全在系统之间的接缝上——闭环恰恰在跨系统处断裂。**

按严重度分 P0 / P1 / P2。

---

## P0 — 闭环真的断了 / 会导致不一致

### P0-1 权限决策"各画各的"，没有单一 PDP 权威
- **现状**：RBAC 图有完整 PDP（`FUNCTION_GATE / MODEL_GATE / ROW_FILTER / FIELD_FILTER / PRECEDENCE / FAIL_CLOSED`）。但 Workflow 图又自带 `FORM_PERMISSION + DATA_SCOPE`；Page 图自带 `DATA_SCOPE + PERMISSION_RENDER`；Data 图自带 `MODEL/ROW/FIELD_PERMISSION + MASK`。行级 / 字段级权限在 4 张图各实现一遍，**没有一条「委托给中央 RBAC PDP」的边**。
- **为什么不合理**：同一种「字段脱敏 / 行过滤」有 4 个实现点，必然漂移。Page 的字段隐藏与 Data 的字段脱敏若判断不一致，谁赢？未定义。
- **改法**：RBAC PDP 定为**唯一决策点（Policy Decision Point）**，其余系统是执行点（PEP），只画「调用 PDP → 拿 ALLOW/FILTER/MASK」的边，不再各自重画权限逻辑。

### P0-2 失效 / 重入闭环（⑥）只在系统内闭合，跨系统是断的
- **现状**：RBAC 有 `EVENT_BUS(role.changed / permission.changed) → INVALIDATION`，但这条事件**没有任何边连到 Workflow / Page / Data 的失效器**。各系统的 `INVALIDATION` 只管自己（Page 的依赖图是 `页面→组件→数据集→api`，不含角色）。
- **为什么不合理**：把 RBAC 的 `manager` 角色改名 / 删掉，**引用它的工作流审批节点、页面可见性配置不会被通知、不会失效**。平台号称一体，但「改一处、连带失效」跨不过系统边界。这是最深的闭环缺陷。
- **改法**：建立**全局依赖图 + 全局失效总线**——`role.changed / model.schema.changed / page.published` 必须能跨系统传播，下游系统订阅并定位受影响产物。（本评审已把它的**可执行版**实现为 Skill 的「跨系统影响分析」，见文末。）

### P0-3 元数据 SSOT 被破坏——字段 / 角色定义在多处
- **现状**：「字段」被三方声明：Data 图 `ENTITY_MODEL / FIELD_MODEL`、Workflow 图 `FORM_TEMPLATE(字段Schema)`、Page 图 `BINDING_SCHEMA / PAGE_SCHEMA`。Workflow 图一边写 `FORM_DESIGNER 唯一输出 Form Schema`，一边又自带 `FORM_TEMPLATE` 的字段 + 权限 Schema，**自相矛盾**。
- **为什么不合理**：一个「请假天数」字段的类型 / 校验 / 权限以谁为准？三份各改各的。
- **改法**：**DataModel 是字段唯一 SSOT**；表单 / 页面只能「引用绑定」数据模型字段，不得各自定义字段。权限 Schema 挂到字段上，不在表单里复制。

## P1 — 一致性 / 契约没对齐

### P1-4 两套角色模型，且继承只在一边
- **现状**：RBAC 有 `Role + AppRole`；App Center 又有 `APP_ROLE(含角色继承)`。但 RBAC 的 `PERMISSION_LOADER(User→Roles→Permissions)` 没有任何继承。
- **问题**：Page 的 `visibleToRoles`、Workflow 的 `assignee=role` 指租户 Role 还是 AppRole？「角色继承」在 App Center 声明却没人在 PDP 解析。
- **改法**：统一一套角色解析（含继承）放 PDP，App Center 只做「成员绑定」，不另立继承语义。

### P1-5 App 是组装根，但它的跨系统校验是空话
- **现状**：App Center 有 `PERMISSION_VALIDATE / DEPENDENCY_VALIDATE`——正是该做跨系统一致性检查的地方（workflow.assignee ∈ app.roles；page.binding ∈ app.dataModels）。
- **问题**：但真实跨引用一条没画进去，validator 停在「名字」层面。
- **改法**：把 App 发布门禁坐实成「跨系统引用闭包校验」——这才是组装根的价值。

### P1-6 App 包的版本钉选（pinning）未定义
- **现状**：Workflow 做对了（`INSTANCE_FACTORY 冻结流程版本与表单快照`）。但 App 运行时 `PAGE_RESOLVER→APP_PAGE→PAGE_DESIGNER` 看似取最新，而 App 又有 `RELEASE_ARTIFACT 依赖快照` 暗示钉版本。
- **问题**：App 发布 v2 后，底层 page 升到 v3，运行时跑哪个版本？未定义 → 行为不可预期。
- **改法**：明确「App 发布即钉死各子产物版本」，运行时按快照解析。

### P1-7 审计 / Trace 碎成 6 份，没有全局关联
- **现状**：6 张图各有 `DECISION_LEDGER + traceId`（Authorization / Workflow / Data / Page / Runtime / AIGC）。
- **问题**：用户「打开 App 页面 → 触发工作流 → 读数据 → 调 AIGC」一个动作，在 5 个库写 5 条 ledger、5 个 traceId，无法端到端串起来。
- **改法**：一个根 traceId 贯穿全链路，各 ledger 带同一根 traceId。

## P2 — 范围 / 命名

### P2-8 数据中台的口径明显比其他 5 张大一号
- Data 图有完整数仓分层 `RAW→ODS→DWD→DWS→ADS` + 联邦查询 + 语义层 + 物化视图——这是 BI / 数仓平台的野心，而其余 5 张是 OLTP 应用配置中台。**更像两个产品**。
- **建议确认**：数据中台是「给应用建表的数据建模器」还是「企业数仓」？两者架构 / 团队 / 成本完全不同，混在一张闭环里会拖累整体。

### P2-9 命名撞车
- AIGC 图的 `SKILL_CONFIG(技能定义)` 与「Skill 能力抽象」是两个东西，文档 / 代码务必区分，否则团队混淆。

---

## 与 Skill 计划的衔接

P0-1（权限单点）、P0-2（跨系统失效）、P0-3（元数据 SSOT）、P1-5（App 跨系统校验）本质都是同一件事：**跨系统引用闭包 + 全局依赖 / 失效分析没人管。**

这恰好是 Skill 框架的「⑥ 跨系统影响分析」能算出来的东西——它是上述架构缺陷的**可执行版本**：给定「改 / 删某个资源」，自动算出全平台受影响的产物。已随本评审实现，见 `client/src/lib/skills/orchestrator.ts` 的 `impact()`。
