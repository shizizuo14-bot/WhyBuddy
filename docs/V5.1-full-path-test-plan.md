# SlideRule V5.1 全路径验收测试文档（Full-Path Acceptance Test Plan）

> **版本**: r1 · 对应架构图 V5.1（修复版）
> **测试面**: `localhost:3000/sliderule`（聊天室 Surface）+ runtime API（vitest）+ 台账回放
> **方法论**: 图即测试 —— 架构图的每一条**边**至少被一个场景遍历一次；每一条**被删除的边 / 禁止的路径**对应一个负向测试（不变式 N1–N6）。
> 覆盖目标 = 全图约 110 条边 + 6 条不变式。

---

## 0. 这份文档怎么用

1. 每个场景（S 系列）包含：**覆盖的边 → 前置状态 → 聊天室操作步骤 → 期望路径 → 机械断言 → 当前可跑状态**。
2. 「机械断言」全部按你的原则写：**二元、可 grep / 可断言、不靠人眼判断**。判定数据来源只有三处：`V5SessionState`（STATE）、`getSessionLedger`/`decisionLedger`（台账）、STATUS 条 DOM。
3. 「当前可跑状态」三档：
   - ✅ **可跑** — 现有代码（含 GCOV/BUDGET/DLEDGER/C-1/C-2 修复）就能执行
   - 🟡 **部分** — 主链可跑，个别断言落空（标注哪条）
   - ⛔ **阻塞** — 被未实现节点挡住（标注挡在哪个节点），先把场景写好放着，节点落地当天即可启用
4. 执行顺序：先跑 **冒烟集**（S1→S2→S4→S10，约 5 分钟），再跑全量。S 系列之间默认**每场景重置会话**，标注「续上一场景」的除外。

## 2. 场景库

### S1 · 冷启动首轮（new_goal 单门入站）✅
**覆盖边**: `CHAT→INTAKE` · `STATE→INTAKE(load+derive)` · `INTAKE→INTERV(分类)` · `INTERV→BUDGET` · `BUDGET→ORCH(放行)` · `ORCH→DLEDGER` · `DLEDGER→T_LEDGER` · `ORCH↔BUS` · `BUS→T_GATE→T_PROV→T_LEDGER→STATE` · `STATE---AWAIT` · `ORCH→STATUS` · `STATE→BOARD` · `BUDGET→T_LEDGER(遥测)`

**前置**: 全新会话（重置会话按钮）。
**步骤**: 聊天框输入任意目标，例「做一个权限管理系统」，发送。
**期望路径**: 消息经单门 INTAKE，分类为 `new_goal`（空态专属），过预算闸放行，ORCH 选能力并落决策账，能力产物经信任脊柱入 STATE，本轮结束停泊 AWAIT。
**机械断言**:
- `controlSignal === "new_goal"`（且仅本轮——下一条消息必须不是 new_goal）
- `decisionLedger.length >= 1`，最新条目含 `{saw, chose, skipped, rationale}` 全字段非空
- 每个产物 `artifact.passedGates` 含 `commit`，`trustLevel ∈ {gated_pass, audited}`
- `runtimePhase === "awaiting"`，`lastTurnId` 与本轮一致
- STATUS 条显示轮次=1、已调用能力数>0
- 台账（ledger 按钮）条目数 === capabilityRuns 数

### S2 · 标准收敛到 clear（灵魂循环第一圈）✅
**覆盖边**: S1 全部 + `ORCH→GCOV` · `CONTRACT→GCOV(分母)` · `GCOV→GOAL(达标写入)` · `GCOV→AWAIT(达标停泊)` · `ORCH→GOAL(只读)`

**前置**: 续 S1，或新会话。
**步骤**: 依次发送「分析安全风险」→「生成可行性报告」（确保合约 required 能力各有一次成功 run、blocking gap 清零）。
**机械断言**:
- 收敛轮 `coverageGate.passed === true`
- `goal.status === "clear"`，且写入仅经 `applyGoalConclusion`（grep 断言：`goal.status` 赋值点全仓只有 `createInitialSessionState` + `applyGoalConclusion` 两处）
- STATUS 条结论徽章显示「已收敛 / clear」且绑定 `sessionState.goal.status`
- **回放断言**（P1 验收原文）: 从台账能列出本会话覆盖了合约哪几项、哪些 gap resolved/waived——写一个 `replayCoverage(state)` helper，输出与合约 required 列表逐项对得上

### S3 · GCOV 硬阻断 + 强制排程 ✅
**覆盖边**: `GCOV→BUDGET(缺能力·强制排程)` · `GCOV` 拒绝分支 · `BUDGET→ORCH` 二次放行

**前置**: 新会话，人造「漏 evidence.search」：合约 required 含 evidence.search，但只发「分析安全风险」（不触发 evidence）。
**步骤**: 直接发「生成可行性报告」（带收敛意图）。
**机械断言**:
- `coverageGate.passed === false`，会话停泊 AWAIT 且标 `partial`，note 含 `[GCOV] blocked`
- `goal.status` 保持 `needs_refinement`（不许跳 clear）
- DLEDGER 有 `GCOV_BLOCKED` rationale 条目
- 缺失能力被强制排回：下一轮 picks 含 `evidence.search`，且该排程**经过 BUDGET**（台账可见预算检查记录在排程之前）
- 补跑后再发收敛意图 → `clear`（衔接 S2 断言）

### S4 · 挑战产物 → 级联失效 → 重新收敛（灵魂循环第二圈，C-1/C-2 验收）✅
**覆盖边**: `BOARD→INTAKE(针对节点)` · `INTAKE→INTERV` · `INTERV→DEP(challenge)` · `DEP→INVAL→STALE→RECOMP` · `RECOMP→BUDGET(重算经预算)` · `STALE→STATE(同步前端)` · 二次 `ORCH→GCOV→GOAL`

**前置**: 续 S2（处于 clear）。
**步骤**: ① 在 risk 卡片点「挑战此结论」（或点 graph 节点，两口等效）；② 观察级联；③ 再发「生成可行性报告」。
**机械断言**:
- 挑战后：risk 及其下游 report 的 id 均 ∈ `staleArtifactIds`（依赖级联），对应 graph 节点 status = `challenged`
- **C-2**: 挑战返回瞬间 `goal.status === "needs_refinement"`（降级经 `applyGoalConclusion`，STATUS 徽章同步变化）
- **C-1**: 下一轮 picks 重新包含 `risk.analyze` **和** `report.write`（stale-only kind 不算存在）
- 重收敛后 `goal.status === "clear"` 第二次成立；新 report 的 `evidenceRefs` 指向**新** risk（不是 stale 旧件）
- **两圈半断言**: 在第二个 clear 上再挑战一次 → 第三圈仍能走通（stale 集合不污染新健康集）
- 卡片挑战与节点点击产生的失效/重排程行为逐字节一致（P2 验收：序列化两路结果 deep-equal）

### S5 · 挑战一条调度决策（DLEDGER → 路由可审）🟡
**覆盖边**: `INTERV(target=Decision)` · DLEDGER 反向边（challenge 指向 decision → gap 重开 → 重排程）

**前置**: 续 S2。
**步骤**: 通过 UI（若有入口）或 runtime API 构造 `UserIntervention{ intent: "challenge", targetDecisionId: <最近一条 DLEDGER id> }` 走 intakeMessage。
**机械断言**:
- 被挑战 decision 对应的 gap 状态从 resolved/waived → 重开（open）
- 触发重排程且经 BUDGET
- 若该 decision 不导致 gap 重开（如挑战一条只记账的 skip），`goal.status` 不降级（C-2 的 ¬条件）
**当前状态**: 🟡 runtime 路径按你 plan 已有 `targetDecisionId`；聊天室 UI 还没有「点一条 DLEDGER 记录发起挑战」的入口——UI 入口缺失先用 API 跑，断言不打折。

### S6 · 预算超限 → 停泊 partial ✅
**覆盖边**: `BUDGET→AWAIT(超限·停泊 partial)` · `BUDGET→T_LEDGER(成本遥测)` · STATUS 预算余量

**前置**: 新会话 + 低限预算 fixture（maxTurns=3）。
**步骤**: 连续发 4 轮普通消息。
**机械断言**:
- 第 4 轮被 BUDGET 拦截：plan 为空、note 含 `blocked_by_budget`、停泊 AWAIT 标 partial
- `goal.status` 不变（保全）
- 台账有该轮预算检查记录（预算 = auditable artifact：能从台账读出当时的限额与用量快照）
- 🟡 STATUS 条「预算余量」「等待原因」展示——若 UI 未实现该字段则此条断言降级为 state 字段断言，UI 补上后升级

### S7 · 预算超限 + 不收敛 → 转人工 ESC ✅
**覆盖边**: `BUDGET→ESC`

**步骤**: 低限预算 + 永不满足合约的构造（required 能力被禁用），驱动到超限且 GCOV 不可满足。
**机械断言**: 状态进入 escalate 分支（`runtimePhase === "failed"` 或显式 ESC 标记），台账留痕。
**当前状态**: 🟡 取决于 ESC 分支是否已实现为独立状态；若现在统一落 partial AWAIT，则先断言 partial + 不可满足标记，ESC 落地后切换。

### S8 · maxRepeat 死循环截断 ✅
**覆盖边**: BUDGET 的 maxRepeat 维度

**步骤**: 构造「同能力无状态变化重复跑」：连续发同一句触发同一能力、期间不产生新 artifact/gap 变化。
**机械断言**: 第 N+1 次重复被截断（picks 不再含该能力或被 BUDGET 拦截），台账记录 `maxRepeat` 命中。

### S9 · 合约早停（够了就停）✅
**覆盖边**: `CONTRACT→BUDGET(早停判据)`

**步骤**: 会话已满足合约（S2 收敛后），继续发与合约无关的泛化消息。
**机械断言**: 出现 `stopped_by_contract_sufficiency` DLEDGER 条目，不再扩张能力调用（picks 为空或仅响应显式新意图），`goal.status` 不变。

### S10 · Commit Gate 打回（验真失败路径）✅
**覆盖边**: `T_GATE→BUS(未过·打回)` · untrusted 产物滞留 · report 拒引用坏上游

**步骤**: 重置会话 → 点「下次让上游失败」→ 发「生成可行性报告」。
**机械断言**:
- risk/counter `trustLevel === "untrusted"`，`gateResults` 含 failed 的 commit
- report 因引用 untrusted 上游自动 gate fail（卡片显示拒绝）
- **C-1 untrusted 变体**: 下一轮（不点失败按钮）发「生成可行性报告」→ picks 必须重新包含 risk/counter/report（untrusted-only kind 不算存在——这是 C-1 修复的 trustLevel 半边）
- GCOV 在该状态下 `passed === false`（`hasTrustedCommittedForCap` 与 pickNext 对 kind 在不在的判断一致）
