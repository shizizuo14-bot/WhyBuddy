# 需求文档:编组(Fabric)流式时间线重构

## 目标

将编组阶段右栏从"8 面板静态工作台"重构为 MiroFish 风格的"流式时间线",实现自动推进、摘要优先、递进式输出的交互体验。用户在选完路线后只需观察系统自动推进,不再需要手动点击各种生成按钮。

## 需求

### 需求 1:纵向时间线布局

系统应将编组阶段的 8 个子阶段渲染为一条纵向时间线,每个节点只有三种状态:
- **已完成**(completed):折叠为一行标题 + 状态 badge + 3 个关键指标,可点击展开
- **活跃**(active):展示当前进度信息,可能有加载动画
- **未来**(future):灰色标题占位,不渲染任何内容

验收标准:
- 时间线从上到下按 `RAIL_SUB_STAGE_ORDER` 顺序排列
- 已完成节点占据最小高度(一行摘要 + 指标行)
- 活跃节点占据主要视觉空间
- 未来节点只显示标题,视觉权重最低
- 整体可滚动,活跃节点自动滚入视口

### 需求 2:自动推进机制

系统应在每个子阶段完成后自动触发下一阶段的 API 调用,无需用户手动点击:
- SPEC 树生成完成 → 自动触发规格文档生成
- 规格文档生成完成 → 自动触发效果预演
- 效果预演完成 → 自动触发提示词包生成
- 提示词包完成 → 自动触发运行时能力注册
- 运行时能力完成 → 自动触发工程落地
- 工程落地完成 → 自动触发产物记忆归档

验收标准:
- 用户选完路线后,整个编组流程自动推进到底
- 任何一步失败时停止推进,在活跃节点展示错误信息
- 用户可以手动点击"重试"恢复推进
- 不会重复触发已完成的阶段

### 需求 3:摘要优先展示

已完成的子阶段不再展示完整工作台面板,只展示精简摘要:
- 一行标题 + 完成状态 badge
- 3 个关键数字指标(复用 `deriveSubStageSummary`)
- 可选的"查看详情"入口(跳转 /specs 或 inline 展开)

验收标准:
- 已完成节点的高度不超过 80px
- 指标数字使用 mono 字体,视觉突出
- "查看详情"不是默认展开,需要用户主动点击
- 详情展开不影响时间线其他节点的位置

### 需求 4:活跃节点进度展示

当前活跃的子阶段应展示实时进度信息:
- 规格文档生成:逐个文档的完成状态(requirements ✓ / design ✓ / tasks ⟳)
- 效果预演:预演数量和当前状态
- 其他阶段:通用的"处理中"状态 + API 路径

验收标准:
- 活跃节点有明确的视觉区分(边框颜色/背景色)
- 加载状态有 spinner 或 skeleton 动画
- 进度信息实时更新(通过 W1 refetch 或 SSE)
- 错误状态有红色提示 + 重试按钮

### 需求 5:流式输出动画

新出现的时间线节点应有流式输出的视觉效果:
- 新完成的节点从活跃态切换到完成态时有 collapse 动画
- 新的活跃节点出现时有 fade-in + slide-up 动画
- 时间线自动滚动到新的活跃节点

验收标准:
- 动画时长 200-300ms,不影响操作流畅度
- `prefers-reduced-motion` 下降级为无动画
- 滚动行为使用 `scrollIntoView({ behavior: "smooth" })`

### 需求 6:保留数据层不变

重构只改 UI 渲染层,不改数据获取和状态管理:
- `useAutopilotRightRailData` hook 保持不变
- `resolveRailSubStage` 纯函数保持不变
- `sub-stage-summary.ts` 摘要派生保持不变
- 后端 API 不新增任何接口

验收标准:
- 现有 262 个右栏测试中,数据层相关测试全部通过
- hook 的 retry / cache / stale policy 机制不受影响
- `/specs` 全宽页面的 `BlueprintProgressPanel` 不受影响

### 需求 7:详情入口保留

用户如果需要深度编辑(修改 SPEC 树节点、评审文档等),应有明确入口:
- 已完成的 SPEC 树节点:点击"查看详情"跳转到 /specs 页面
- 已完成的规格文档:点击"查看详情"跳转到 /specs 页面
- 不在时间线内嵌完整 workbench

验收标准:
- 每个已完成节点有"查看详情"链接
- 链接跳转到 /specs 页面对应的 workbench
- /specs 页面保持完整编辑能力不变
