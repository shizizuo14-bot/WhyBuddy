# 需求文档

## 简介

本规格定义 SPEC 自动驾驶链路的第 1 步：用户输入执行目的、想法或一个或多个 GitHub 链接后，系统需要将这些输入规范化为项目入口资产，为后续澄清、路线生成和 SPEC 树推导提供统一上下文。

这个阶段的目标不是直接执行任务，而是把模糊输入整理成可追踪、可复用、可回放的项目起点。

## 术语表

- **InputEntry**：一次输入会话中的原始输入记录，可能包含文本、GitHub 链接、附件或补充说明
- **GitHubSource**：被解析的 GitHub 仓库、分支、提交或路径上下文
- **IntakeRecord**：经过归一化后的项目入口记录
- **ProjectContext**：后续澄清、路线生成和 SPEC 推导所依赖的项目上下文

## 需求

### 需求 1：支持多入口输入

**用户故事：** 作为用户，我希望直接输入执行目的、想法文本或 GitHub 链接，以便快速开启一个项目推演会话。

#### 验收标准

1.1 系统 SHALL 支持纯文本输入、单个 GitHub 链接和多个 GitHub 链接混合输入。  
1.2 系统 SHALL 支持从现有项目中继续导入上下文，而不是强制新建项目。  
1.3 系统 SHALL 将输入来源保留为可追踪的原始记录。  
1.4 系统 SHALL 为无效输入提供可理解的错误提示，而不是静默失败。

### 需求 2：解析 GitHub 上下文

**用户故事：** 作为系统，我希望能够解析 GitHub 仓库上下文，以便自动抽取项目背景、目录结构和关键文件信息。

#### 验收标准

2.1 系统 SHALL 识别仓库 URL、分支、commit、目录路径和 issue / pr 链接。  
2.2 系统 SHALL 抽取仓库元信息，包括仓库名、默认分支、README、语言分布和目录骨架。  
2.3 系统 SHALL 对重复链接进行去重，并保留来源顺序。  
2.4 如果仓库不可访问或权限不足，系统 SHALL 保留失败原因并继续允许用户补充文本上下文。  
2.5 系统 SHALL 记录每个 GitHubSource 的解析状态和时间戳。

### 需求 3：生成归一化项目入口

**用户故事：** 作为系统，我希望把输入和 GitHub 上下文归一化成统一项目入口，以便后续流程可以只依赖一个 ProjectContext。

#### 验收标准

3.1 系统 SHALL 为每次输入生成一个 IntakeRecord。  
3.2 系统 SHALL 将输入目标、GitHubSource、初始摘要和来源证据合并为 ProjectContext。  
3.3 系统 SHALL 自动关联当前项目或创建新项目，且不丢失原始来源。  
3.4 系统 SHALL 为后续澄清流程提供 projectId、goal、summary 和 sourceIds。  
3.5 系统 SHALL 允许用户再次打开同一入口并查看完整来源历史。

### 需求 4：沉淀输入资产

**用户故事：** 作为项目资产系统，我希望保存输入入口及其解析结果，以便未来可以重放、审计和继续推演。

#### 验收标准

4.1 系统 SHALL 将 IntakeRecord 写入项目资产存储。  
4.2 系统 SHALL 保存原始文本、GitHubSource 列表和解析摘要。  
4.3 系统 SHALL 支持后续流程按 projectId 查询输入入口。  
4.4 系统 SHALL 为输入记录提供可引用的 evidenceId 或 sourceId。  
4.5 系统 SHALL 保持输入资产与后续澄清、路线和 SPEC 的关联关系。
