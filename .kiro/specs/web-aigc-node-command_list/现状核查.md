# `web-aigc-node-command_list` 现状核查

## 核查结论

当前主仓已经具备 `command_list` 的最小可用闭环，本轮完成后可按 `4/4` 视为完成。

本轮实现重点不是把 `command_list` 接进运行时热文件，而是在不修改 `server/index.ts`、`server/core/workflow-runtime-engine.ts` 的前提下，先完成以下闭环：

1. 有结构化候选命令列表输出；
2. 能对接现有 `nl-command` 能力；
3. 能输出兼容 `selection` / HITL 的桥接结构；
4. 能记录命令生成与选择事件。

## 代码事实

### 1. 候选命令列表结构

文件：
- `shared/nl-command/command-list.ts`
- `server/routes/node-adapters/command-list-node-adapter.ts`

当前已定义：
- `CommandListCandidate`
- `CommandListSnapshot`
- `CommandListEvent`
- `CommandListSelectionBridge`

候选命令包含：
- `candidateId`
- `label`
- `commandText`
- `description`
- `recommended`
- `source`
- `commandTarget`
- `clarificationPreviewTarget`

其中：
- `commandTarget` 指向现有 `/api/nl-command/commands`
- `clarificationPreviewTarget` 指向现有 `/api/nl-command/clarification-preview`

### 2. 与任务命令中心的对接

文件：
- `shared/nl-command/api.ts`
- `server/routes/nl-command.ts`

本轮新增接口：
- `POST /api/nl-command/command-list/generate`
- `POST /api/nl-command/command-list/:listId/select`

能力：
- `generate` 用于生成候选命令列表；
- `select` 用于记录用户或系统最终命中的候选命令；
- 两者都复用了现有 `nl-command` 主路由，无需改 `server/index.ts`。

### 3. 与选择节点的联动

文件：
- `shared/nl-command/command-list.ts`
- `server/routes/node-adapters/command-list-node-adapter.ts`

本轮通过 `selectionBridge` 复用现有 HITL / selection 结构：
- `selectionBridge.nodeType = "selection"`
- `selectionBridge.decision` 直接产出 `MissionDecision`
- 候选命令被映射成 `MissionDecision.options`
- 若存在推荐项，还会返回 `recommendedSubmission`

选中结果会回传：
- `metadata.nodeType = "command_list"`
- `metadata.branchKey = candidateId`
- `metadata.interactionId`
- `metadata.formData.selectedCandidateId`

这使它可以自然接入已有任务决策与选择提交流程。

### 4. 命令生成与选择事件

文件：
- `shared/nl-command/command-list.ts`
- `server/routes/node-adapters/command-list-node-adapter.ts`

当前事件类型：
- `generated`
- `selected`

当前存储方式：
- 内存 `eventStore`
- 内存 `snapshotStore`

本轮目标是“最小可用闭环”，所以这里采用轻量内存实现，已经足够支撑：
- 测试验证
- 路由闭环
- 结构化留痕
