# 需求文档:输入步骤流式时间线重构

## 目标

将输入步骤(当前时间线第 1 步"输入")从"表单堆叠"模式重构为流式时间线,与编组阶段保持一致的交互范式。用户在一条纵向时间线上完成:目标输入 → intake 创建 → 澄清 → 路线生成 → 路线选择,每一步自动推进,只在关键决策点需要用户介入。

## 需求

### 需求 1:输入步骤拆分为 5 个子阶段时间线

系统应将输入步骤内部拆分为 5 个子阶段,按纵向时间线渲染:
1. **目标输入**(target_input):填写执行目标 + GitHub 地址
2. **输入记录**(intake):创建 intake,归一化来源
3. **澄清**(clarification):生成澄清问题 + 用户回答
4. **路线生成**(route_generation):自动生成 RouteSet
5. **路线选择**(route_selection):展示候选路线,用户选择

验收标准:
- 5 个子阶段按顺序从上到下排列
- 每个子阶段有三态:completed / active / future
- 已完成的子阶段折叠为一行摘要 + 关键指标
- 活跃子阶段展示交互内容(输入框/问题列表/路线卡片)
- 未来子阶段灰色占位

### 需求 2:自动推进机制

系统应在每个子阶段完成后自动推进到下一步:
- 用户填完目标并点击"创建输入记录" → 自动进入 intake 阶段
- intake 创建成功 → 自动生成澄清问题
- 澄清就绪(readiness.status === "ready") → 自动生成路线
- 路线生成完成 → 自动展示路线选择
- 用户选择路线 → 自动进入编组阶段

验收标准:
- intake 创建成功后自动触发澄清生成
- 澄清就绪后自动触发路线生成(已实现)
- 路线生成后直接展示选择卡片(已实现)
- 选择路线后时间线第 1 步整体变为"完成"

### 需求 3:活跃子阶段的交互内容

每个活跃子阶段展示对应的交互 UI:
- **目标输入**:执行目标 textarea + GitHub 地址 textarea + "创建输入记录"按钮
- **输入记录**:来源数量 / 重复数量 / 项目上下文指标
- **澄清**:问题列表 + 回答输入框 + "提交澄清"按钮 + 就绪度指示
- **路线生成**:"正在生成路线..."进度提示
- **路线选择**:候选路线卡片列表 + 选择按钮

验收标准:
- 目标输入阶段保留现有 textarea 交互
- 澄清阶段保留现有问题列表 + 回答输入框
- 路线选择阶段保留现有 RouteOption 卡片
- 所有交互逻辑不变,只改布局和视觉呈现

### 需求 4:已完成子阶段的摘要展示

已完成的子阶段折叠为精简摘要:
- **目标输入完成**:目标文本前 50 字 + GitHub 链接数
- **intake 完成**:来源数 / 重复数 / 项目上下文状态
- **澄清完成**:就绪度百分比 + 已回答/总问题数
- **路线生成完成**:候选路线数
- **路线选择完成**:已选路线标题

验收标准:
- 每个已完成子阶段高度不超过 60px
- 指标使用 mono 字体
- 可点击展开查看详情(可选)

### 需求 5:与编组时间线视觉一致

输入步骤的时间线应复用编组阶段的 `TimelineNode` 组件:
- 相同的三态图标(✓ / 🔄 / ○)
- 相同的 badge 样式(完成/进行中)
- 相同的 fade-in 动画
- 相同的自动滚动行为

验收标准:
- 复用 `TimelineNode` 组件,不新建独立组件
- 视觉风格与编组时间线完全一致
- 动画行为一致

### 需求 6:保留现有数据流不变

重构只改 UI 渲染层:
- 所有 API 调用函数保持不变
- 所有 useState / useCallback 保持不变
- `handleCreateIntake` / `handleGenerateClarifications` / `handleSaveAnswers` / `handleGenerateRouteSet` / `handleSelectRoute` 保持不变

验收标准:
- 现有 AutopilotRoutePage.test.tsx 测试通过
- 数据流和 API 调用逻辑零改动
