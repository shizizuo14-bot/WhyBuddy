## 当前结论

`web-aigc-node-web_qa` 已完成最小可用闭环，当前顶层任务可按 `4 / 4` 认定完成。

本次实现采用“复用底座”的方式完成，而不是新造一套网页问答系统：

1. 复用 `open_page` 节点能力来规范化网页/链接目标。
2. 复用 `rag web-aigc document-search` 契约承接网页或文档检索候选。
3. 复用 `chat` 节点能力生成最终回答。
4. 当网页检索不可用时，可回退到 `knowledgeService.query` 的知识问答底座。

## 已落地能力

### 1. 输入输出结构已定义

已新增共享契约 `shared/web-qa/contracts.ts`，明确以下结构：

- 节点类型：`web_qa`
- 输入：
  - `question`
  - `pages`
  - `search`
  - `knowledgeFallback`
  - `workflowId / sessionId / missionId / agentId / stage`
- 输出：
  - `answer`
  - `reply`
  - `citations`
  - `sourceLinks`
  - `evidenceList`
  - `fallbackUsed`
  - `fallbackReason`
  - `metadata`

### 2. 网页内容获取适配器已成立

当前 `web_qa` 的最小网页上下文来源支持两类：

- 内联网页上下文：
  - 通过 `pages[]` 直接传入 `title / summary / content / href / route`
- 检索式网页上下文：
  - 通过 `search.scope + search.options` 调用现有 `documentSearch` 执行器获取候选结果

链接目标会复用 `open_page` 适配器做统一归一化，因此：

- 内部路由
- 外部 URL
- 页面标题 / pageId / openMode

都能在 `sourceLinks` 中稳定输出。

### 3. 来源链接与证据已输出

当前输出中已经提供：

- `citations`
  - 适合直接给回答区或结束节点展示
- `sourceLinks`
  - 适合前端渲染“打开网页/文档”入口
- `evidenceList`
  - 适合下游继续处理、回放或审计

证据来源区分为：

- `page`
- `search`
- `knowledge_fallback`

### 4. 失败回退逻辑已验证

当以下情况发生时，系统可回退到知识问答底座：

- 网页搜索执行器抛错
- 网页检索结果为空且启用了 `knowledgeFallback`
- 只有兜底知识可用

回退后会明确返回：

- `status: fallback`
- `strategy: knowledge_fallback`
- `fallbackUsed: true`
- `fallbackReason`

若既没有网页上下文，也没有启用知识回退，则返回失败态，避免静默成功。

## 测试结论

已覆盖并通过以下场景：

1. 基于内联网页 + 检索结果生成回答，并输出引用、来源链接、证据。
2. 网页检索异常时回退到知识问答底座。
3. 无上下文且无回退配置时返回失败。
4. 路由层非法 `nodeType` 返回 `400`。
5. 路由层成功返回完整 `web_qa` 输出结构。
6. 路由层知识回退返回 `fallback` 状态。
