# web-aigc-node-open_report 现状核查

## 结论

当前仓库已经补齐 `open_report` 的最小闭环，且这次实现优先复用了现有 `workflow report / department report download / replay` 体系，没有新造独立报告系统。

当前可成立的最小闭环是：

- 节点输入统一为 `reportType + workflowId/replayId + managerId + format + openMode`
- 服务端能够解析目标报告或回放
- 能输出标准化“可打开目标”
- 在启用 permission engine 时可以走 `api:call` 权限校验
- 已有 adapter 测试与 route 测试覆盖成功、拒绝、缺失、参数错误等关键路径

本轮实现的重点不是“直接替前端执行打开动作”，而是“把 open_report 节点标准化为一个可被前端消费的打开目标解析器”。

## 已落地能力

### 1. 统一输入输出

已新增：

- `server/routes/node-adapters/open-report-node-adapter.ts`

当前支持的输入目标：

- `final_report`
- `department_report`
- `replay`

当前标准化输出包含：

- `status`
  - `completed`
  - `denied`
  - `not_found`
- `reportType`
- `mode`
  - `view`
  - `download`
- `resource`
- `target`
  - `href`
  - `apiHref`
  - `uiHref`
  - `downloadHref`
- `context`
- `governance.permission`

这意味着 `open_report` 节点现在可以作为一类独立节点，被统一调用后返回“应该打开哪里”。

### 2. 报告定位与链接生成

当前复用的现有后端能力：

- `GET /api/workflows/:id/report`
  - final report 查看
- `GET /api/workflows/:id/report/download?format=json|md`
  - final report 下载
- `GET /api/workflows/:id/report/department/:managerId/download?format=json|md`
  - department report 下载
- `GET /api/replay/:missionId`
  - replay 元数据校验
- `/replay/:missionId`
  - 前端 replay 页面路径

因此当前 `open_report` 能返回三类最小目标：

- final report 查看目标
- department report 下载目标
- replay 页面打开目标

### 3. 权限与校验

当前已支持：

- 必填字段校验
  - `reportType`
  - `workflowId`
  - `managerId`
  - `replayId`
- workflow 存在性校验
- report / replay 可用性校验
- 在挂接 `permissionEngine` 时执行：
  - `resourceType = api`
  - `action = call`
  - `resource = GET /api/...`

当前实现方式与仓库里其他最小节点 adapter 一致，属于“先标准化资源，再走统一权限引擎”的做法。

### 4. 最小 route 闭环

已新增：

- `server/routes/open-report.ts`

当前入口为：

- 节点级独立入口：`POST /api/workflows/open-report/nodes/execute`

本轮收口聚焦于“open_report 节点自身的 route + adapter + tests + spec 对账”，不把 `server/routes/workflows.ts` 中与其它 runtime 热区混在一起的改动纳入本次提交。

HTTP 状态映射为：

- `completed -> 200`
- `denied -> 403`
- `not_found -> 404`
- 输入错误 -> `400`

## 测试结论

依据：

- 已有统一输出 `target + context + governance`

### 2. 对接 workflow 报告接口

可以勾选。

依据：

- 已直接复用 `workflows.ts` 的 final report / department report 接口
- 已复用 replay route 进行回放可达性校验

### 3. 增加权限校验

可以勾选。

依据：

- 已在 adapter 中支持 `permissionEngine.checkPermission(...)`
- 资源按 `GET /api/...` 方式统一构造

### 4. 验证前端打开链路

可以勾选，但应限定为“前端可消费的打开目标链路已验证”。

依据：

- replay 目标已对齐现有 `/replay/:missionId`
- report 目标已对齐现有 `/api/workflows/.../report*`
- route 测试已验证目标能被节点执行入口稳定返回

限制说明：

- 当前还没有做一个专门的前端 `open_report` 组件去自动跳转或弹窗
- 现阶段是“节点输出可打开目标”，由前端决定如何消费

## 当前边界

当前未做的部分：

- 没有新增独立 `open_report` 页面 UI
- 没有做复杂的“报告预览弹层 / 内嵌浏览 / 多报告切换器”
- 没有引入额外的工作区级或租户级报告可见域模型
- 未把 `server/routes/workflows.ts` 的主线挂接改动纳入本次节点切片提交

这次的设计选择是合理的，因为 `open_report` 作为 UI 动作节点，优先目标应该是：

- 与已有报告/回放体系对齐
- 提供稳定可消费的最小输出
- 不侵入其他 lane 的前端主线

## 后续建议

如果后续要继续扩展，可以按这个顺序补：

1. 在前端增加 `open_report` 结果消费器
   - 根据 `target.kind / href / uiHref / downloadHref` 决定打开方式
2. 给 final report 增加专门的前端查看页
   - 避免只走原始 JSON API
3. 把 open_report 纳入更高层 runtime 节点编排入口
   - 形成和 `chat / knowledge / mcp` 类似的统一节点接入面
4. 视需要补更细的权限资源
   - 如 `workflow_report`
   - `department_report`
   - `replay_view`
