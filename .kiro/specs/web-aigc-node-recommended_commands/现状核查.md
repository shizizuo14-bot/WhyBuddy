# 现状核查：推荐命令节点

## 核查范围

- `shared/nl-command/api.ts`
- `shared/nl-command/contracts.ts`
- `server/routes/nl-command.ts`
- `server/routes/node-adapters/recommended-commands-node-adapter.ts`
- `server/tests/nl-command/routes.test.ts`
- `server/tests/nl-command/recommended-commands-node-adapter.test.ts`

## 当前结论

`web-aigc-node-recommended_commands` 已具备“最小可用闭环”，可以按 `recommended_commands` 节点的口径提供推荐命令列表、供人工选择/确认的结构，以及最小审计落点。

## 已落实能力

### 1. 推荐命令结构已落地

- `GET /api/nl-command/plans/:id/suggestions` 不再是 501，占位接口已改为真实返回。
- 返回体包含：
  - `suggestions`
  - `selectionOptions`
  - `confirmPayload`
  - `observability`
- 单条 `suggestion` 已补充：
  - `source`
  - `rationale`
  - `recommendedCommand`
  - `selectionOption`
  - `confirmOption`

### 2. 已接入现有 suggestion 能力

- `server/routes/node-adapters/recommended-commands-node-adapter.ts` 优先复用 `DecisionSupportEngine` 的：
  - `suggestCostOptimization`
  - `suggestResourceAdjustment`
- 当主仓未注入 `decisionSupportEngine` 或当前 plan 不可用时，会自动退化为启发式推荐，避免接口再次退回不可用状态。

### 3. 已与 selection / confirm_judge 打通最小联调口径

- `selectionOptions` 可直接映射到 `selection` 的候选项。
- `confirmPayload` 已给出确认 prompt、默认推荐项和 `suggestionId` 分支键，可直接映射到 `confirm_judge`。
- 这里实现的是“数据结构联调闭环”，不是 runtime 主链自动编排；符合当前 lane 不改 `server/core/workflow-runtime-engine.ts` 的约束。

### 4. 推荐与采纳事件已可追踪

- 推荐生成时写入 `suggestion_generated`
- 采纳时写入 `suggestion_applied`
- 采纳接口 `POST /api/nl-command/plans/:id/apply-suggestion` 已返回：
  - `adjustment`
  - `appliedSuggestion`
  - `audit.suggestionAppliedEntryId`

## 仍然存在的边界

以下能力当前仍属于“最小可用”，不应夸大为完整版：

- 当前没有把 `recommended_commands` 自动接入 workflow runtime 主链，只提供了可被主线接线消费的数据与接口。
- `apply-suggestion` 的计划更新目前是局部/最小口径：
  - 若存在 plan，会返回最小更新后的 `updatedPlan`
  - 若 suggestion 本身没有实体级 `changes`，则只返回采纳结果与审计信息
- 默认挂载的 `nl-command` 路由使用无依赖的 `RecommendedCommandsAdapter`，因此在线上主仓未注入 orchestrator / decision-support 时，会走启发式推荐而非完整计划推理。
