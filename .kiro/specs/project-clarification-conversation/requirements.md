# 需求文档：Project Clarification & Conversation

## 目标

将现有“信息不足 -> 补全面板”的逻辑升级为项目内澄清和上下文沉淀机制。澄清不是临时弹窗，而是项目 spec 和路线规划的来源。

## 需求

### 需求 1：项目内对话历史

系统应为每个项目保存独立对话历史，包含用户输入、系统补问、用户回答、关键决策和状态更新。

### 需求 2：澄清问题结构化

系统应将澄清问题结构化保存，包括问题、原因、影响范围、回答类型、是否必答和默认建议。

### 需求 3：澄清结果合并

用户回答澄清后，系统应将结果合并到项目上下文，并标记对应来源。

### 需求 4：澄清驱动 spec

澄清问答应成为 spec 生成或更新的来源，而不是仅用于一次性 mission 创建。

### 需求 5：避免过早执行

当项目目标、约束、范围、技术栈、交付物或风险信息不足时，系统应优先澄清，而不是直接启动深度执行。

### 需求 6：支持渐进式澄清

系统应支持多轮澄清，但每轮问题数量应克制，优先问影响路线和执行边界的问题。

### 需求 7：记录关键决策

当用户在澄清中做出关键选择时，系统应记录为 project decision，并可在 spec 和 route 中引用。

### 需求 8：澄清可跳过

用户应能选择跳过部分澄清，但系统需要记录风险和默认假设。

### 需求 9：澄清状态可视化

Project Cockpit 应能展示当前缺失信息、已确认信息和下一轮建议。

### 需求 10：兼容现有补全面板

改造应复用现有 clarification dialog / panel 的交互基础，只改变数据归属和项目上下文承接。
