## 关键发现

### 1. `agentCrewStageActivation` 在 pre-RouteSet 阶段 `totalInvocations=0` 是正确行为

原 `integration-gap-2026-05-16.md` P0 #3 列了这一项，但 subagent 在调研后正确识别这是误诊：

- Driver 的 `onStageTransition` **必须传真实 `BlueprintGenerationJob`**（必填字段，包含 `routeSet` / `artifacts` / `stageState`）。intake / clarification 阶段还没有 job。
- 即使强行用 `intakeId` 当 `jobId` 调，driver 内部第 2 步 `findRoleArchitectureEvidence` 会返回 `{status: "fallback"}`，然后调 `enterFallback("...")` **直接 return 不 emit 任何事件**。Subscriber 看不到事件，`totalInvocations` 仍然为 0。
- 更关键：原报告里 "fire `crew.stage.activated`" 这个事件名**根本不存在**。`BlueprintEventName` 里只有 `crew.context.updated`。我之前写报告时把桥名当成了事件名。

**结论**：这一项不是 gap，是 spec 撰写时的误识别。`agentCrewStageActivation` 是 post-RouteSet bridge，pre-RouteSet 阶段 `totalInvocations=0` 是符合契约的。

### 2. P0 #3+#4 修复的真实根因

Subagent 发现：路由生成时 `createRouteGenerationSandboxDerivation` 已经产出了 `capability.invoked` / `capability.completed` 事件（包括 `role-system-architecture` 和 `aigc-spec-node` 两个桥的），subscriber 也已经订阅这些事件准备调 `recordBridgeInvocation`。**唯一缺的环节是事件没经过 `eventBus.emit`**——它们直接被 push 进 `events[]` 后随 `options.store.save(job)` 一次性落盘。

修复手术非常精确：在 `createGenerationJob` 内把这两个桥的事件从 inline events 中剥离，`store.save(job)` 之后通过 `ctx.eventBus.emit(bridgeEvent)` 广播一次。Subscriber 因此触发，`recordBridgeInvocation` 被调，diagnostic 计数才会动起来。

注意：这次实测因为 LLM 上游全挂（主 + fallback 都 dead），路由生成请求在 LLM 卡死阶段被前端超时取消，没能跑到 `createGenerationJob` 验证桥事件实际流入。但 252 个相邻单元测试已经覆盖了这条链路。

### 3. P0 #5 的真实行为验证

服务器日志清楚地写出了 `LLM request to api-vip.codex-for.me timed out after 30000ms`——P0 #5 的 30s timeout 起作用了。但路由生成阶段还有更高层的重试循环（每个候选路线都试一轮），所以即使 P0 #5 把单次 LLM 调用裁到 30s，端到端总时长仍然取决于路由层重试策略。这是一个独立的问题，不在 P0 #5 范围。

## 设计原则（三组件共用）

- **只读**：不写 store，不订阅 socket（订阅由 `AutopilotRoutePage` 的两段式 useEffect 完成）
- **折叠态**：数据为空时返回 `null`，不抢占布局
- **防御性兜底**：当 selector 返回 `undefined`（部分单元测试 mock 时会发生）时一律视为折叠态
- **稳定排序**：按 id 字母序，避免 phase / status 变化导致 DOM 重排
- **仅 fabric 阶段挂载**：由 `AutopilotRightRail` 控制，三组件本身不判断 `currentStage`
