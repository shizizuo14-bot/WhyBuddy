# 需求文档：自动驾驶目的地输入

## 目标

把当前“输入指令/创建任务”的入口升级为“输入目的地/启动任务自动驾驶”的入口。用户输入内容后，系统必须先展示自己理解的目的地、缺失信息、可进入的路线规划状态，而不是直接把文本提交成普通 mission。

## 与现有 18 份 task-autopilot specs 的引用关系

本 spec 不重新定义 Destination、Route、Drive State、Takeover 等概念，直接引用并前端落地以下既有 specs：

- `task-autopilot-platform-positioning`
- `task-autopilot-core-concepts`
- `destination-model-and-parser`
- `route-planner-and-route-model`
- `mission-model-to-autopilot-model-mapping`
- `autopilot-cockpit-information-architecture`

## 当前差距

- 用户不知道输入什么内容才能触发自动驾驶。
- 输入框仍带有“命令/任务提交”心智，目的地结构不明显。
- 目的地解析结果没有在提交前成为可确认对象。
- 缺失信息只表现为后续澄清，不够像导航里的“缺少目的地信息”。
- 附件、约束、成功标准、时间线对路线规划的影响没有被即时解释。

## 需求

### 需求 1：入口必须明确引导用户输入目的地

系统必须将入口文案、placeholder、示例和空状态统一为“目的地输入”心智。

验收口径：

- 输入框默认文案必须包含目标、交付物、约束、时间线、成功标准。
- 用户不需要阅读 README 就能知道如何触发自动驾驶。
- 输入为空时必须展示可点击的目的地示例。

### 需求 2：输入后必须生成目的地预览

系统必须在提交前展示解析出的目的地摘要。

验收口径：

- 至少展示目标、交付物、约束、时间线、成功标准五类字段。
- 未识别字段必须标为“待补充”，不能静默隐藏。
- 目的地预览必须能进入后续路线规划面板。

### 需求 3：系统必须解释缺失信息

当输入不足以规划路线时，系统必须显示缺失原因和补充建议。

验收口径：

- 缺少目标、交付物、时间线、成功标准、约束时应有独立提示。
- 系统必须区分“可先规划”与“必须先补路标”。
- 缺失提示应能驱动 clarification，而不是只做静态文本。

### 需求 4：附件必须成为目的地上下文的一部分

添加附件后，入口必须解释附件将如何影响路线规划。

验收口径：

- 附件数量、类型、解析状态必须可见。
- 附件会导致深度路线、高级编排或补充路标时，系统必须说明原因。
- 附件错误不能阻断用户理解当前目的地状态。

### 需求 5：入口必须支持从旧 mission/task 语义平滑迁移

系统应兼容现有 `commandText`、`sourceText`、`mission` 创建链路。

验收口径：

- 旧任务创建仍可工作。
- 新入口提交时必须能把 `selectedRouteId`、目的地解析摘要传入后续链路。
- 不允许为了前端心智升级而破坏已有 NL Command / Workflow submit 行为。
