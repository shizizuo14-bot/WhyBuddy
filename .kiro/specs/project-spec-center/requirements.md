# 需求文档：Project Spec Center

## 目标

建立项目 spec 中枢，让项目从一次性问答升级为可演化、可版本化、可引用、可验证的规格体系。Spec 是 FSD 路线规划和深度执行的主要依据。

## 需求

### 需求 1：每个项目有当前 Spec

系统应允许每个项目拥有一个当前 spec，并支持从项目目标、澄清回答和执行证据生成第一版。

### 需求 2：Spec 版本化

系统应支持 spec 版本历史，记录每次变更来源、摘要、时间和替代关系。

### 需求 3：Spec 来源可追溯

Spec 应记录来源消息、澄清回答、证据、artifact、GitHub 调研结果或用户决策。

### 需求 4：Spec diff

系统应能展示相邻版本之间的变化摘要，帮助用户理解项目是如何演化的。

### 需求 5：Spec 完整度

系统应评估 spec 完整度，包括目标、用户、范围、约束、交付物、技术栈、验收标准和风险。

### 需求 6：Spec 驱动路线

FSD Route Planner 应优先基于当前 spec 生成主路线、备选路线和保守路线。

### 需求 7：执行结果反哺 Spec

任务执行产物、调研结论、失败原因和用户接管决策应能生成 spec 更新建议。

### 需求 8：Spec 可编辑或可确认

用户应能确认、编辑或要求系统重写当前 spec；确认后的 spec 才适合进入深度自动驾驶。

### 需求 9：轻量 Markdown 先行

第一阶段 spec 可采用结构化 Markdown，不要求复杂富文本或 block editor。

### 需求 10：项目沉淀入口

Project Cockpit 和任务中心应能进入当前项目的 Spec Center。
