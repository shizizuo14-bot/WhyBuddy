# 需求文档：Project Evidence, Artifact & Replay

## 目标

建立项目级证据、产物和回放闭环。系统不只是回答或执行，还要把为什么这么做、做了什么、产出了什么、如何演化沉淀回项目。

## 需求

### 需求 1：产物挂项目

文档、SVG、代码、报告、截图、原型、spec diff 等产物应通过 `projectId` 归档到项目。

### 需求 2：证据挂项目

日志、运行事件、用户决策、路线选择、资料来源、失败原因和接管记录应保存为 project evidence。

### 需求 3：产物来源可追踪

每个 artifact 应能追踪来源 mission、route、spec、message 或 evidence。

### 需求 4：Spec 来源可追踪

Spec 版本应能引用 evidence 和 artifact，说明当前规格为什么这样演化。

### 需求 5：Replay 可按项目查看

用户应能按项目查看执行时间线、关键决策、路线变化、任务状态和产物生成过程。

### 需求 6：证据服务接管

当用户需要终止、重试、调整路线或确认关键选择时，系统应展示足够证据支撑决策。

### 需求 7：SVG 架构图项目化

系统生成或维护的 SVG 架构图、路线图、进度图应作为 project artifact，而不是散落在全局 docs 心智中。

### 需求 8：执行结果回流项目

任务完成、失败或取消时，应把结果摘要、日志、产物和下一步建议回写项目。

### 需求 9：证据分层展示

首页只展示摘要，任务中心展示执行证据，Replay 展示完整轨迹，避免首屏过重。

### 需求 10：兼容现有 evidence / replay 能力

改造应复用现有审计、replay、任务日志、artifact 和 runtime event 基础。
